## 分野の全体像

### ドキュメントの種類と研究量

| 種類 | 研究量 | 主な知見 |
|------|--------|----------|
| インラインコメント | 多い | コメントあり→欠陥発見約14%速い |
| 識別子名 | 中程度 | コメントより理解への影響が大きいという研究あり |
| docstring / メソッドレベルコメント | 中程度 | コード要約（Code Summarization）と接続 |
| README / アーキテクチャ文書 | 少ない | 保守コストとの関連が語られるが実験的検証は少ない |
| API ドキュメント | 少ない | 利用者視点の理解支援として重要だが研究は限られる |

### 主要サーベイ

- **"A Decade of Code Comment Quality Assessment: A Systematic Literature Review"** (JSS 2023)
  - 2,353論文を分析，47論文をレビュー
  - 21の品質属性を特定（ほとんどの研究は4属性しか扱っていない）
  - 主な属性：整合性（コードとの一致），完全性，明確性，簡潔性
  - 自動品質評価はいまだ困難。人手評価が主流
  - 対象言語のほとんどがJava（他言語への一般化が不明）

### 未解決課題（初回把握時点）

- コメントの自動品質評価が困難
- Java以外の言語・ドメインへの一般化
- 短い人工スニペットでの実験が多く，実際のコードベースへの適用が不明

## コメント（Code Comment）研究

### セッション記録：2026-04-17 — 第1回

### コメントの種類

| 種類 | 説明 |
|------|------|
| インラインコメント | コードと同行，特定の文を説明 |
| ブロックコメント | 複数行，ロジックのまとまりを説明 |
| docstring / メソッドコメント | 関数・クラスレベルの正式なドキュメント |
| TODO / FIXME / HACK | 技術的負債の自己申告（SATD） |
| ライセンス・ヘッダコメント | 著作権・ファイルレベル |

統一タクソノミーは存在しない。Steidl & Hummel (ICPC 2013) が基礎的な分類を提案。

---

### コメント品質の属性

Rani et al. (JSS 2023) が21属性を特定。ほとんどの研究は以下の4つしか扱っていない：

| 属性 | 内容 |
|------|------|
| 一貫性 (Consistency) | コードの実際の動作と一致しているか（最多研究） |
| 完全性 (Completeness) | 必要な箇所に十分なコメントがあるか |
| 整合性 (Coherence) | コメントとコードの論理的な流れが合っているか |
| 有用性 (Usefulness) | 読み手の理解に実際に役立つか |

---

### コメントは理解を助けるか

**助ける場合：**
- 視線追跡実験（EMSE 2025）：コメントは読者の視線を誘導し，複雑なコード理解を有意に改善
- 読み方のパターン：「コメント優先」と「コード優先」の2種が確認される
- ビーコン（手がかり）として機能し，ナビゲーションを助ける

**助けない・害になる場合：**
- **古くなったコメント（Comment Rot）**：コードと食い違うコメントはバグ混入を**1.5倍**増やす（ICPC 2019）
- **過剰なコメント**：多すぎるとかえって可読性を下げる
- **明白なコメント**：コードを読めば分かることを再記述しても意味がない
- **誤情報は無情報より悪い**：開発者はコードよりコメントを信頼する傾向があるため

---

### Comment Rot（コメントの腐敗）

- Fluri et al.：**コード変更の97%はコメント変更と同一コミットで行われる**
- しかし**新規追加コードにはコメントがつかないことが多い**
- 8システム中6システムで90%のケースは共進化するが，残り10%に問題が蓄積

**JIT（Just-In-Time）不整合検出ツール：**

| ツール | 手法 | 性能 |
|--------|------|------|
| Panthaplackel et al. (AAAI 2021) | DLモデルでコミット前に不整合を検出 | 複数ベースラインを上回る |
| CoCC | 学習ベースの不整合検出 | 精度90%以上 |
| CARL-CCI | 構造化diffを利用，コードアクションでアノテーション | 高精度 |

---

### SATD（Self-Admitted Technical Debt）

TODO / FIXME / HACK 等を「技術的負債の自己申告」として研究する分野。

- オープンソース開発者の47%がSATDアノテーションを使用（産業界では71%が使わない）
- SATDのあるメソッドはサイズが大きく，複雑度が高く，可読性が低い
- バグ混入率が高く，コードレビューで承認される確率が**6〜7%低い**
- 2025年：LLM生成コードがSATDを増加させるという報告が登場

---

### コメント品質の自動評価：なぜ難しいか

- 「良いコメント」の普遍的定義がない
- 読み手のスキル・タスク・ドメインによって品質が変わる
- コードと違いコンパイル・実行時に検証されない
- MLによるコメント臭検出：F1 ≈ 0.77
- BLEU等の自動指標はMETEORが最も人手評価と相関するが代替には不十分（r ≈ 0.7）

---

### 研究の構造

```
コメント研究
├── 効果研究（コメントあり vs なし，視線追跡・タスク成績）
├── 品質研究（属性の定義・測定，コメント臭）
├── 共進化・整合性研究（Comment Rot，JIT検出ツール）
└── SATD研究（TODO/FIXME等の分布・影響，LLM生成SATDへの拡張）
```

### 「コメントが必要なコード」の特定に向けた議論

#### 問いの構造

「良いコメント」を定義するには「どのコードにコメントが必要か」が先決。
情報付加性（コメント情報量 − コードから読める情報量）を測るには，右辺の定義が必要。

#### 「コメントのみ追加コミット」アプローチ

コードを変更せずコメントだけ追加するコミット → 開発者が「説明が必要」と判断したコードのラベルとして使えるという仮説。

**評価：**

| 観点 | 内容 |
|------|------|
| 新規性 | 高い。直接的な先行研究が存在しない |
| 実現可能性 | 高い。GitHubからの大規模マイニングは標準的手法（MSR分野） |
| サンプル数 | Fluri et al. より97%のコメント変更はコード変更と同一コミット → コメントのみコミットは稀 |
| 動機の多様性 | コードレビュー指摘・スタイルガイド準拠・混乱起因などを区別できない |

**動機の多様性への対処案（2026-04-20 実験結果を踏まえた更新）：**

1. コミットメッセージで絞り込む（"clarify complex logic", "document tricky behavior" 等）。ただしこの信頼性自体も検証が必要。全体の11.8%（4,533件）がヒット。
2. **issueと紐づいたコミットに限定する**（`fixes #123` 等の明示的参照）→ **実装・実験済み**。

   **実験結果（500リポジトリ・38,541件）：**

   | フィルタ | 件数 | 割合 |
   |---|---|---|
   | 全コメントのみコミット | 38,541 | 100% |
   | `#` を含む（issue候補） | 5,436 | 14.1% |
   | issue番号抽出成功 | 4,612 | 12.0% |
   | issue内容が難解さ関連（`issue_complexity=1`） | 593 | 1.5% |

   **判明した主要ノイズ：**
   - `#NNN` がissueではなくPR番号を指しているケースが多数（GitHub APIの `pull_request` 属性で判別可能）
   - `TheAlgorithms/Java` 等の教育目的リポジトリでは「JavaDoc追加PR」が大量にあり偽陽性を生む
   - 真陽性の例：`ghidra#6498`「`getLength()` がビット単位かバイト単位か不明」，`guava#2178`「`Cache.stats()` の使い方が文書化不足」

   **PR番号フィルタ適用後（2026-04-20）：** `filter_prs.py` で593件を再確認 → **247件**に絞り込み（約58%がPR参照の偽陽性）。

   **247件のissue内容分析（2026-04-20）：** 228件の固有issueを全件精読。難解さの根本原因として以下が判明：
   1. セマンティクスが名前/型から読めない（単位・解釈の曖昧さ）
   2. 前提条件・制約が隠れている（呼び出し順・スレッドセーフ性・例外）
   3. 設計の意図・なぜそうなっているかが不明
   4. 名前と動作の乖離

   実質的に難解さ起因のissue：約210件（85%）。残り35件は機能追加・バグ報告の偽陽性。

3. **PRレビューコメント起因**：「レビュアーが混乱を指摘 → 著者がコメントのみコミットで対応」という因果が取れれば動機が最も明確。検出条件はレビューコメントのタイムスタンプ＜コミットのタイムスタンプ，かつ同一ファイルの変更。ただし以下の理由から**実現可能性に懸念あり**：
   - botとの区別は可能だが，人間による実質的なレビューはプロジェクト規模に強く依存する
   - 「コードが難解」を指摘するレビューコメントはバグ指摘・実装変更要求より少数派
   - issueフィルタ後の593件でさらに絞ると，サンプル数が研究として成立しない可能性がある

#### 関連する先行知見

- **Fluri et al.**：コード変更の97%はコメント変更と同一コミット → コメントのみコミットは稀
- **Panthaplackel et al. (AAAI 2021)**：JIT不整合検出の訓練データにコメント変更コミットを使用（コメントのみはスコープ外）
- **Hu et al. (ICSE 2022)**：92%の実務者が「複雑なメソッド」へのコメントを支持，91%が「自明でないメソッド」へのコメントを支持
- **Arafati & Riehle (ICSE 2009)**：OSSのコメント密度の適正範囲は30〜60%

#### 「情報付加性」の研究上の位置づけ

「情報付加性」という名前では研究されていないが，以下として間接的に扱われている：
- コメント臭「明白なコメント」（情報付加性の否定的ケース）
- 意味的類似度指標 STSS・STASIS（類似度が高すぎる＝情報付加性が低い）
- 「有用性」属性（最も操作化が難しい属性として記述，測定手法未確立）
